## Reporting a Vulnerability

We take security vulnerabilities seriously. If you discover a security issue, please follow these steps:

### Do NOT

- Open a public GitHub issue
- Discuss the vulnerability publicly
- Exploit the vulnerability

### Do

1. **Email us directly** at security@platxa.com with:
   - Description of the vulnerability
   - Steps to reproduce
   - Potential impact
   - Any suggested fixes (optional)

2. **Use responsible disclosure**:
   - Allow us reasonable time to fix the issue (typically 90 days)
   - We'll acknowledge receipt within 48 hours
   - We'll provide updates on our progress

### What to Expect

- **Acknowledgment**: Within 48 hours of your report
- **Initial Assessment**: Within 7 days
- **Resolution Timeline**: Depends on severity
  - Critical: 7-14 days
  - High: 14-30 days
  - Medium: 30-60 days
  - Low: 60-90 days

### Recognition

We appreciate security researchers who help us keep our users safe:

- Credit in our security advisories (if desired)
- Acknowledgment in release notes
- Consideration for our bug bounty program (when available)

## Security Measures

### Input Validation

All user inputs are validated and sanitized:

```typescript
import { sanitizeInput, validateInput } from "platxa-frontend-agent"

// Sanitize user input
const safe = sanitizeInput(userInput)

// Validate with detailed results
const result = validateInput(userInput)
if (!result.isValid) {
  console.error(result.errors)
}
```

### CSS Injection Prevention

Generated CSS is validated against injection attacks:

```typescript
import { validateCSS, sanitizeCSS } from "platxa-frontend-agent"

const result = validateCSS(generatedCSS)
if (!result.isValid) {
  // Use sanitized version
  const safeCSS = result.sanitized
}
```

### Secrets Detection

Code is scanned for accidentally included secrets:

```typescript
import { detectSecrets, hasSecrets } from "platxa-frontend-agent"

if (hasSecrets(generatedCode)) {
  const secrets = detectSecrets(generatedCode)
  // Handle detected secrets
}
```

## Best Practices for Users

### Environment Variables

Never hardcode secrets in your code:

```typescript
// BAD
const apiKey = "sk-abc123..."

// GOOD
const apiKey = process.env.API_KEY
```

### Content Security Policy

When deploying generated components, use appropriate CSP headers:

```html
<meta http-equiv="Content-Security-Policy" content="
  default-src 'self';
  script-src 'self';
  style-src 'self' 'unsafe-inline';
  img-src 'self' data: https:;
  font-src 'self' https://fonts.gstatic.com;
">
```

### Dependencies

Keep dependencies updated:

```bash
# Check for vulnerabilities
npm audit

# Fix automatically where possible
npm audit fix
```

## Security Checklist for Contributors

When contributing code, ensure:

- [ ] No hardcoded secrets or API keys
- [ ] User inputs are sanitized
- [ ] CSS outputs are validated
- [ ] No eval() or dynamic code execution
- [ ] Dependencies are from trusted sources
- [ ] Tests cover security edge cases

## Known Security Considerations

### AI-Generated Code

Code generated by AI agents may contain security issues. Always:

1. Review generated code before use
2. Run security linting (`npm run lint`)
3. Test with security scanners
4. Validate in staging before production

### Third-Party Dependencies

This package uses third-party dependencies. We:

- Regularly update dependencies
- Run `npm audit` in CI/CD
- Monitor for security advisories
